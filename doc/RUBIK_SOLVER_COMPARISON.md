# 魔方求解实现对比与优化方向

本文对比当前 MagicCubeSolver 的几个求解路径，并说明它们与常见 IDA* / PDB / Kociemba 思路的关系。旧版本中“IDA* 主要是弱启发式完整空间搜索”的描述已经不再完整。

## 1. 当前求解路径

| 算法 | 当前状态 | 适用场景 | 主要局限 |
|------|----------|----------|----------|
| 反向移动 | 仅在明确选择 `reverse-moves` 且存在打乱历史时使用 | 本应用内打乱后的演示 | 不能求解外部输入状态 |
| Kociemba | 使用 `kociemba-wasm`，通用快速路径 | 用户只关心尽快解开 | 依赖 wasm 库与 cubestring 契约 |
| Thistlethwaite | 自研四阶段表驱动求解器 | 观察群论分阶段还原 | 首次构建表较慢，非全局最短 |
| IDA* | 浅层完整空间 IDA* + 随机打乱分阶段 IDA* | 自研搜索路径、教学和对比 | 分阶段解非全局最短，首次表构建较慢 |

## 2. 与外部资料的关系

公开资料中的高性能 3x3 IDA* 通常不会只靠简单错块计数，而是结合：

- 模式库（PDB）或专用抽象距离表。
- 紧凑状态编码。
- 分阶段或子群约束。
- 大量预计算与缓存。

当前项目的分阶段 IDA* 和 Thistlethwaite 已经走向这一类“拆小问题 + 查表启发”的路线，但仍是浏览器内轻量实现：表在运行时按需构建，尚未做离线 PDB 文件、Web Worker 预热或磁盘缓存。

## 3. 为什么同叫 IDA*，速度会差很多

IDA* 只是搜索框架，性能主要由启发式和状态空间决定：

- 完整空间 + 弱启发式：随机 3x3 很容易爆炸。
- 完整空间 + 强 PDB：可行性显著提升，但表生成和存储成本高。
- 分阶段 + 抽象距离表：牺牲全局最优性，换取浏览器中更实用的求解时间。

本仓库当前 UI 的 `IDA*` 属于第三种路线：浅层仍保留完整空间搜索，随机打乱走分阶段搜索。

## 4. 当前优化优先级

| 优先级 | 方向 | 说明 |
|--------|------|------|
| 高 | Web Worker 预热表 | 减少首次求解时 UI 卡顿和等待感 |
| 中 | 表构建进度提示 | 让用户知道第一次慢是在初始化缓存 |
| 中 | 表持久化或离线生成 | 减少刷新后的重复构建成本 |
| 低 | 追求全局最短 IDA* | 工程量大，更接近独立求解器项目 |

## 5. 建议的产品策略

- 默认实用求解：优先 Kociemba。
- 自研算法展示：Thistlethwaite 和 IDA* 都已可用于随机打乱，但应标明“分阶段、非全局最短”。
- 性能体验优化：优先处理首次表构建的可见进度和 Worker 化，而不是继续调完整空间 IDA* 参数。

## 6. 当前验证方式

当前测试不是只检查求解器返回非空数组，而是会执行返回步骤并验证结果回到已解 cubestring：

```text
applyMovesToCubestring(start, solution) === SOLVED_CUBESTRING
```

这使得算法、转动语义、cubestring 编解码可以被同一套断言覆盖。
