# 中英文切换（i18n）方案说明

本文档面向当前仓库（React 18 + Vite + TypeScript），总结一套可落地的中英文界面切换方案，便于后续迭代时按步骤实施。

**文件位置**：仓库根目录下 `doc/I18N_ZH_EN_PLAN.md`。

---

## 1. 现状与目标

| 项目 | 说明 |
|------|------|
| **现状** | UI 文案多为硬编码；`index.html` 中 `lang` 固定为 `zh-CN`；无国际化库。 |
| **目标** | 用户可在中文、英文之间切换；切换后即时生效；刷新或再次打开页面可记住选择（可选）。 |

---

## 2. 方案选型（简要对比）

| 方案 | 优点 | 缺点 |
|------|------|------|
| **react-i18next + i18next** | 生态成熟、文档全、支持命名空间/懒加载、与 React 集成好 | 依赖略多，学习成本中等 |
| **自定义 Context + JSON** | 零依赖、完全可控 | Plural、插值、懒加载需自研 |
| **Lingui / FormatJS** | 编译期优化、类型友好 | 配置与构建链更重 |

**推荐**：在本项目中优先采用 **react-i18next + i18next**。与 Vite 配合成熟，社区示例多，后续若增加第三种语言或拆分模块（命名空间）成本较低。

---

## 3. 推荐架构

### 3.1 资源组织

```
src/
  i18n/
    index.ts              # 初始化 i18next、注册语言包
    locales/
      zh-CN.json          # 中文（或 zh-CN/common.json 若拆命名空间）
      en.json             # 英文
```

- 键名建议 **扁平或短层级**，如 `cube.title`、`solver.error.invalid`。
- 长段落可单独键，避免单键内嵌整段 HTML；若必须富文本，可用 Trans 组件或拆成多条。

### 3.2 运行时行为

1. **默认语言**：例如 `zh-CN`（与当前产品主语言一致）。
2. **用户选择**：通过设置 `i18n.changeLanguage('en')` 切换。
3. **持久化（推荐）**：将选择写入 `localStorage`（键如 `app.locale`），初始化时读取并 `changeLanguage`。
4. **浏览器语言（可选）**：首次无缓存时，用 `navigator.language` 与 `en` / `zh-CN` 做前缀匹配，再决定是否默认英文。

### 3.3 与 React 的集成

- 在应用根部（如 `main.tsx`）在渲染前 `await` 或同步完成 `i18n` 初始化（若使用 HTTP 后端则需异步；**本地 JSON 导入可同步初始化**）。
- 使用 `I18nextProvider` 或 `initReactI18next` 提供的 `useTranslation`；组件内 `t('key')` 取文案。
- 切换语言后，依赖 `t` 的组件会随 i18n 事件重渲染，无需手动刷新整页。

### 3.4 `index.html` 与无障碍

- 切换语言时同步更新 `document.documentElement.lang`（如 `zh-CN` / `en`），与屏幕阅读器、字体行为一致。
- 若未来做 SEO 多语言，再考虑按路由或子域拆分；纯 SPA 可先只做运行时切换。

---

## 4. UI 层面的语言切换入口

- **位置**：页眉/设置面板/页脚任选一处固定入口；移动端可用下拉或图标菜单。
- **状态**：当前语言高亮；仅两种语言时可用切换按钮（中 ⇄ En）。
- **范围**：所有面向用户的字符串逐步从硬编码迁到 `t()`；错误提示、按钮、空状态、帮助文案一并纳入。

---

## 5. 实施步骤（建议顺序）

1. 安装依赖：`i18next`、`react-i18next`（版本与 React 18 兼容即可）。
2. 新建 `src/i18n/index.ts`，合并 `zh-CN`、`en` 资源并 `init`。
3. 在入口文件引入 `./i18n`（保证先于根组件执行）。
4. 选一个页面或组件试点，把文案替换为 `t('...')` 并补齐两个 JSON。
5. 实现语言切换控件 + `localStorage` 持久化。
6. 在应用挂载后设置 `document.documentElement.lang`。
7. 其余页面与工具函数中的用户可见字符串分批迁移；**纯算法、魔方记号（如 U/R/F）一般不做翻译**，除非产品明确要求。

---

## 6. 注意事项

| 主题 | 建议 |
|------|------|
| **类型安全** | 可用 `as const` + 键类型，或 `i18next` 的类型插件，减少拼写错误。 |
| **插值** | 使用 `t('key', { name: value })`，避免字符串拼接。 |
| **复数** | 英文复数规则用 i18next plural；中文常可共用一条。 |
| **Three.js / 画布内文字** | 若 3D 场景内有文字纹理或 DOM overlay，切换语言时需同步更新数据源或重绘。 |
| **包体积** | 两语言 JSON 通常很小；若未来语言增多，可用 `import()` 按语言拆 chunk 懒加载。 |
| **测试** | E2E 可固定一种语言跑；或对切换控件做冒烟测试。 |

---

## 7. 小结

采用 **react-i18next**，以 **JSON 语言包 + 根初始化 + `useTranslation`** 为主线，配合 **localStorage** 与 **`document.lang`**，即可在现有 Vite React 项目中稳定实现中英文切换。实施时采用「先基础设施、再试点页面、最后全量迁移」的节奏，风险可控。

若后续需要服务端渲染或按路由分语言，可在本方案基础上扩展 URL 前缀或独立构建目标，不必推翻当前设计。
