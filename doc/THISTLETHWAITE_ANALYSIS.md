## 2. 理论上的 Thistlethwaite 四阶段（简述）

经典 Thistlethwaite 将魔方群逐步缩小到子群，每一阶段只允许特定操作集合，目标状态是**子群内的合法约束**（而非一步到位“全块归位”）：

| 阶段 | 常见目标（表述因教材略有差异） | 典型允许操作思路 |
|------|-------------------------------|------------------|
| G0→G1 | 边块**朝向**满足（边朝向进入子群） | 常限制为不含破坏该性质的 U/D 四分之一转等 |
| G1→G2 | 角块**朝向** + 边块在**正确层**（U/D 与中间层）等 | 如 F,B,R,L,U²,D² 等 |
| G2→G3 | 角块、边块**位置**进一步受限 | 如仅 F²,B²,R²,L²,U,D 等 |
| G3→G4 | 仅用 180° 转动完成还原 | F²,B²,R²,L²,U²,D² |

要点：**每一阶段的“目标函数”必须与该阶段子群定义一致**；若把“角块全部归位”误当成 G1→G2 的目标，则与标准 Thistlethwaite 不等价，且可能在当前允许操作下**根本不可达**。

## 3. 当前实现的分阶段逻辑（代码层面）

实现位置：`src/utils/thistlethwaite.ts`。

### 3.1 阶段 0→1（边块朝向）

- `isInG1()` **恒为 `true`**，即**完全跳过** G0→G1。
- 注释说明为简化：未实现边朝向判定。

**影响**：若初始状态在真实意义下不在 G1，后续在受限操作下可能**无解**或需要额外步数，与标准算法前提不一致。

### 3.2 阶段 1→2（注释写「角块方向」，代码写「角块位置」）

- 目标函数 `isInG2(state)`：比较每个角块的 **`coordinate` 是否与已解状态一致**（即**角块位置全部归位**）。
- 允许操作：`G1_MOVES`（F,B,R,L 全角度 + U²,D²）。

**与理论的偏差**：标准 G1→G2 通常关心**角块朝向**（及边层等），而不是“八个角块 ID 全部回到 home 位置”。后者是**更强**的条件，且在给定子群生成元下**不一定可达**。因此 BFS 可能**不存在**到达该目标的序列 → 表现为阶段 1→2 **失败或返回空解**。

### 3.3 阶段 2→3（边块位置）

- 目标函数 `isInG3(state)`：所有边块 **`coordinate` 与已解一致**（边块位置全部归位）。
- 允许操作：`G2_MOVES`。

若上一阶段目标定义错误，进入本阶段的状态可能已偏离真实 Thistlethwaite 轨道；再叠加下文 **§4.1** 的搜索缺陷，极易超时或判无解。

### 3.4 阶段 3→4（仅用 180°）

- 目标：`isInG0`（与 `isInG0` 命名相反，此处表示**全盘已解**）。
- 允许操作：`G3_MOVES`。

若前序阶段未把状态带入正确的 G3 子群，本阶段也可能无法在搜索深度/时间内到达已解。

## 4. 导致“算不出来”的核心问题

### 4.1 【致命】`stateKey` 使 visited 失效，BFS 几乎无法展开

实现：`stateKey` 对**所有角块坐标字符串排序**、对**所有边块坐标字符串排序**后拼接。

在合法 3×3 魔方状态中：

- 八个角块**始终占据**八个角坐标；
- 十二条边块**始终占据**十二条边坐标。

因此对「坐标多重集合」排序后的结果，在**任意**合法状态下都相同——**与角块/边块的置换、扭转无关**。

后果：

- `visited` 集合用该 key 去重时，会把**大量本质不同的状态**当成同一状态；
- BFS 会错误地剪枝，导致**几乎搜不到有效路径**（除起点即目标等平凡情况）。

这是当前 Thistlethwaite **最可能**导致“完全算不出来”的实现 bug。

**正确方向**：去重键必须编码**置换与朝向**，例如：

- 按固定 ID 顺序拼接每个 cubie 的坐标 + 朝向（或颜色在 cubie 局部轴上的编码）；或  
- 使用已用于其它求解器（如 IDA*）的**规范状态编码**。

### 4.2 阶段目标与经典 Thistlethwaite 不一致

- G0→G1 被跳过；
- G1→G2 使用「角块位置全对」而非「角块朝向（及边层）」等标准目标。

即使修复 `stateKey`，若目标仍与允许操作生成的子群不匹配，仍可能出现**无解搜索**或**深度爆炸**。

### 4.3 搜索参数与复杂度

- `solveCube` 传入 `maxDepthPerStage = 5`（再在各阶段 `+2`、`+4` 等微调）。
- BFS 有 `MAX_NODES`、`timeout`（如阶段 2→3 为 30s）。

对错误目标或错误剪枝，浅深度 + 节点上限会表现为**频繁超时**或**空解**。

### 4.4 失败时的返回值不一致

- 阶段 2→3 失败：**抛出异常** → `solveCube` 捕获后提示换算法。
- 阶段 1→2 或 3→4 部分失败：**返回 `[]`**。

用户可能看到「无步数 / 无效果」或错误提示混用，调试时容易混淆。

## 5. 可行解决方案（按投入产出排序）

### 5.1 必做：修复状态编码与 visited

- 实现**与 cubie 模型一致**的完整 `stateKey`（含置换与扭转）。
- 验证：对已解状态做随机 `applyMove` 若干步，BFS 在极小深度下应能回到已解（在目标函数正确的前提下）。

### 5.2 二选一：对齐理论或改名

**方案 A（对齐 Thistlethwaite）**

- 实现 G0→G1 边朝向检测与阶段搜索（或查表）。
- 按文献实现各阶段目标（角朝向、边层、位置等）与允许转动集合。

**方案 B（不宣称 Thistlethwaite）**

- 将菜单与文档改为「受限移动 BFS 演示」等名称，避免与标准四阶段算法混淆。

### 5.3 工程清理

- 已重命名：`cubeSolver.ts` 中 `solveByCubeSolverKociemba(cubestring)`（避免与自研 Thistlethwaite 同名）。

### 5.4 产品侧建议

在修复完成前，UI 已标注「开发中」是合理的；也可在失败时**统一**提示：优先使用 Kociemba、反向打乱或 IDA*。

## 6. 小结

| 问题 | 严重程度 | 说明 |
|------|-----------|------|
| `stateKey` 仅用排序后的坐标 | **致命** | 合法状态下 key 几乎不变，visited 误判，BFS 失效 |
| 跳过 G0→G1 | 高 | 与标准算法前提不符 |
| G1→G2 目标用角块位置代替朝向/子群条件 | 高 | 目标可能不可达或极难 |
| 深度/节点/超时限制 | 中 | 在目标正确时可调参 |
| 与自研同名的 cube-solver 入口（已改为 `solveByCubeSolverKociemba`） | 低 | 已降低混淆 |

**结论**：当前「算不出解」的首要技术原因是 **`stateKey` 设计错误导致搜索被错误剪枝**；其次才是阶段定义与经典 Thistlethwaite 不一致。修复状态编码后，再根据是否需要「真·Thistlethwaite」决定阶段目标的完整重写或产品上的算法更名。
